Shipped 2026-07-08: server-side customer search replacing the old client-side TanStack `globalFilter` (which only filtered the visible 25-row page). RPC `list_customers_page` gained `p_query` in a prior unit (commit `7da043f`).

**Files:** `src/lib/data/customer.ts` (`listCustomersPage` takes `query?: string`, joins the `"use cache"` key automatically — no extra cache-key wiring needed, Next 16 Cache Components key on all function params), `src/app/(app)/customers/actions.ts` (`fetchCustomersPage(cursor, query?)` normalizes: trim, cap 100 chars, empty→undefined), `src/components/customers/use-customers-search.ts` (new — the search state machine as a hook, not inlined in the list client), `src/components/customers/customers-search.tsx` (new — controlled input, `useId()` per instance since it's rendered twice, desktop+mobile), `src/components/customers/customers-table.tsx` (pure presenter now, `customers-filter.ts` deleted), `src/components/customers/customers-mobile-list.tsx` (owns its own empty-state via `activeQuery` prop, mirrors the table's pattern).

**Why a separate hook file:** `CustomersListClient` was already at the 200-line ESLint function cap before adding search; the search state machine (debounce, out-of-order request-id guard, browse↔search transition) extracted cleanly into `useCustomersSearch` with an inversion-of-control API (`onClear`/`onSearchResult` callbacks let the hook drive *when*, the list-client owns *what* rows/cursor mean).

**Why:** Feeds future customer-list or similar debounced-search work.
**How to apply:** Gotchas worth remembering before writing another debounced-search-with-race-guard hook:
1. `eslint-plugin-react-hooks`'s `react-hooks/refs` rule (React Compiler-era) blocks **any** `ref.current` read/write during the render body — even ones that look like the sanctioned "adjust state during render when a prop changes" pattern (which is fine for `useState` setters, not for refs). Ref bookkeeping tied to a prop-driven reset must move into an effect keyed on the same trigger; the state resets stay synchronous in render to avoid a stale-value flash.
2. When the "adjust state during render" pattern needs to reset state that lives inside a **child hook** (not the calling component itself), duplicate the same pattern one level down: give the hook its own `useState` mirror of the resync key (`committedResyncKey`) and compare inside the hook. React's synchronous re-render-on-mid-render-setState cascades correctly through hook boundaries (confirmed: 3 render passes, no visible flicker, hook call order unchanged across passes — Rules of Hooks intact).
3. **`useEffect` is too late for ref invalidation that guards against in-flight promises — use `useLayoutEffect` instead.** First pass deferred the `searchRequestIdRef`/`activeQueryRef` bump to a plain `useEffect`, which the reviewer caught as a real race (2026-07-08, see [[project-customers-search-staleness-races]] in reviewer memory): `useEffect` is scheduled post-paint, which is *later* than any microtask, so an already-in-flight fetch's `.then()` (a microtask) queued at the moment of resync runs and reads the stale-not-yet-bumped ref before the effect ever fires. `useLayoutEffect` runs synchronously in the same commit as the render-phase state reset, before the JS engine drains the microtask queue — closes the gap completely. This codebase already has an established `useLayoutEffect` precedent (`src/components/invoices/use-invoice-create-form.ts`), so no SSR-warning workaround was needed.
4. **A staleness guard must be unconditional across every async path that mutates the same rows, not just the ones "obviously" in scope.** First pass only checked `requestId` staleness `if (queryAtDispatch)` — i.e., only for search-triggered fetches — leaving plain browse-mode load-more/pagination completely unguarded. The reviewer caught that a slow browse fetch dispatched right before the user starts typing a search would resolve later and append onto the (now-search) rows. Fix: capture `requestId` and check it unconditionally for every dispatch path (browse is just `query === undefined` through the same machinery); keep query-specific side effects (like the result-count increment) separately gated.

Out-of-order protection: a single `searchRequestIdRef` bumped on every new search dispatch, clear-to-browse, AND resync (via `useLayoutEffect`); any async response — search, load-more, or pagination-next, browse or search — checks `searchRequestIdRef.current === requestId` (captured at dispatch) before applying, unconditionally. Load-more/pagination-next reads `activeQueryRef.current` fresh (not React state) at await-time to avoid stale-closure bugs from the 300ms debounce window.

Zod pattern: prefer a dedicated `z.string().trim().max(N).optional()` schema over hand-rolled `trim()`/`slice()` normalization, even for a soft "collapse invalid to undefined" case (no hard reject) — mirrors the existing `SearchQuerySchema` shape in the same actions file. Don't extend/mutate an existing schema used by an unrelated action just to avoid a second declaration; a sibling schema with a documented reason ("same shape, different semantics") is safer than chaining onto shared state.

**Accessibility-auditor pass (2026-07-08), 3 High + 1 Medium, all fixed:**
5. **Never let a shared "isLoading" wrapper dim/disable a sibling control that's still meant to be interactive.** `customers-table.tsx`'s `isLoading && "pointer-events-none opacity-60"` wrapper originally wrapped the search input too, so the exact moment the user is typing (mid-fetch) the input text dropped to ~2.55:1 contrast — the one guaranteed time a real user is looking at it. Fix: split into two sibling divs — the search row stays outside the dimmed wrapper (always full opacity, always interactive); only the DataTable/empty-state/pager region gets `bodyRef` + the loading dim. Check this pattern anywhere a pulse/dim/pointer-events-none wrapper wraps more than the thing that's actually still-loading.
6. **First-pass fix for duplicate `role="status"` regions overcorrected — collapsing to ONE node broke the mobile browse-mode announcement.** Original bug: 3 live regions fired together on search settle, one factually wrong at zero results. First fix collapsed to a single shared `statusText` using the desktop `from`/`to` page-slice formula for the browse branch — but mobile never advances `pageIndex` (it appends via Load More, not paging), so after one Load More mobile showed 50 cards while AT heard "Showing customers 1–25". Re-verify caught this as a regression. **Correct fix:** keep the search/empty branches shared (they're already phrased off `customers.length`, not a page slice, so no viewport-specific bug there) but split the *browse-mode* text into two `role="status"` nodes gated by the existing `max-mobile:hidden`/`min-mobile:hidden` breakpoint utilities — CSS `display:none` removes the non-matching one from the accessibility tree, so still only one is ever announced, but each viewport gets a formula that matches its actual pagination model (desktop: `from`–`to` slice; mobile: `customers.length` running total). Lesson: when consolidating multiple live regions into one, check whether the text-generating formula itself is viewport-agnostic before assuming one node covers both — "de-duplicate the live region" and "use one shared formula" are separable decisions.
7. **A row-swap (not just an unmount) can silently drop focus to `<body>` — and both rendered surfaces sharing the same underlying row state need their own containment check, not just the one that was top-of-mind.** `handleSearchResult`/`handleClearSearch` replace all rows in `customers` state, which feeds BOTH the desktop `DataTable` and the mobile card list off the same array — first pass only wrapped `tableBodyRef` around the desktop subtree, so a search settling while focus was on a mobile card link still dropped to `<body>`. Fix: `mobileListRef` around the mobile card-list container (sibling to, and excluding, the mobile `CustomersSearch` instance) and OR both containment checks (`isFocusInside(mobileListRef.current) || isFocusInside(tableBodyRef.current)`) before the swap; both breakpoints restore to the same `tableTopRef` (always rendered, valid focus target regardless of viewport). Lesson: a fix scoped to "the table" in a component that renders parallel desktop/mobile surfaces off the same state needs an explicit callout of which surface(s) it covers — the auditor re-verify is what caught the missing mobile half.
8. Threading `isPending`/`isLoading` down to a list's `aria-busy` is cheap and easy to forget for every branch that can render while loading — including an early-return branch (the mobile empty-search-results paragraph), not just the primary `<ul>`.
